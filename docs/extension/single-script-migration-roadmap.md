# 单脚本迁移主线路线图

## 目的

本文件用于明确：
- 当前哪些地基已经够用
- 从哪里开始停止继续抽象“未来多脚本中心”
- 从现在开始如何围绕当前这一个脚本进入扩展迁移主线

## 当前停止线

站在“尽快把当前单脚本迁进扩展”这个目标上，当前已经具备的必要地基是：
- 独立项目根目录与清晰目录边界
- 可本地加载的 MV3 扩展骨架
- 共享设置模型与共享存储层
- background / options / popup 的最小统一消费
- `alibaba-labelx` 的页面规则真源
- 页面识别层
- SPA 路由观察层
- 只读页面状态采集层
- 站点层页面模型 / 页面类型规则 / 状态对象契约统一

结论：
- 地基已经达到“够用”标准。
- 现在不再优先继续做面向未来多平台的抽象。
- 接下来进入单脚本迁移主线。

## 现在可以暂时不做的事

以下内容不是当前单脚本迁移前的必要条件：
- 更复杂的平台注册表
- 更复杂的共享层二次抽象
- 预先为第二个脚本设计通用站点框架
- `content.js` 彻底纯壳化
- query/hash 全覆盖的路由体系
- 排行榜、词库服务端、云端接口前置重构

原则：
- 只有当当前单脚本迁移真的撞到这些边界时，再补最小地基。

## 单脚本迁移主线阶段图

### 阶段 1：只读 DOM 适配层

目标：
- 建立统一 DOM 查询入口
- 为 `task-list` / `task-detail` 提供关键只读探针
- 产出稳定的只读适配结果结构

为什么排在这里：
- 当前页面规则和页面状态对象已经稳定，已经足以支撑 DOM 探针。
- 不补这层，就无法把旧脚本里真正依赖 DOM 的能力安全拆分迁入。

主要涉及：
- `extension/sites/alibaba-labelx/`
- `docs/dom-adapter-plan.md`
- `docs/site-state-contract.md`

明确不做：
- fetch 拦截
- 保存链路
- 自动化动作
- 文本写回

当前状态：
- 已完成首版。
- 当前已经可以稳定返回 `task-list / task-detail / unknown` 的统一只读适配结果。

### 阶段 2：站点上下文提取稳定化

目标：
- 基于 DOM 适配结果，收紧 `taskId / taskTitle / domSignals / contextInfo`
- 让页面状态对象与真实页面结构更稳定对齐
- 保证同一路由下的上下文变化仍可被重新采集和验证

为什么排在这里：
- 旧脚本后续很多能力都依赖“当前任务是谁、当前页面处于什么状态”。
- 这些输入先稳定，后面迁业务才不会把脆弱 selector 扩散到各处。

主要涉及：
- `extension/sites/alibaba-labelx/page-state-collector.js`
- DOM 适配相关文件
- `docs/site-state-contract.md`
- `docs/context-fields-contract.md`

明确不做：
- 保存动作
- 自动提交
- 服务端对接
- `content.js` 的大范围收口
- DOM 适配层统一输出结构改造

当前状态：
- 已完成。
- `taskId / taskTitle / domSignals / contextInfo` 已由 `page-state-collector.js` 按固定优先级回填。
- collector 已支持同一路由下基于页面状态签名的重复采集判断。

### 阶段 3：content 装配层继续收口

目标：
- 让 `content.js` 更接近站点装配入口
- 把局部设置读取与 badge 注入边界进一步收紧

为什么排在这里：
- 先有稳定站点状态和 DOM 适配结果，再收口装配层才不会反复改。

主要涉及：
- `extension/sites/alibaba-labelx/content.js`
- 站点层内部新旧模块衔接

明确不做：
- 高风险业务逻辑
- 复杂共享层重构

当前执行边界：
- 默认仍限定在 `extension/sites/alibaba-labelx/` 内推进。
- 不为了外移局部设置读取而直接把 content runtime 改造成 shared 存储层消费者。

当前状态：
- 已完成。
- `runtime-gate.js` 已承接局部设置读取与平台启用门禁。
- `runtime-debug.js` 已承接 badge 注入与 runtime 调试对象暴露。
- `content.js` 已收紧为初始化、门禁、顺序编排和启动入口。

### 阶段 4：低风险业务能力迁移

目标：
- 优先迁移只读或低副作用的页面辅助能力
- 例如信息展示、页面辅助识别、只读统计、低风险 UI 辅助

为什么排在这里：
- 这类能力最容易验证，也最不容易破坏平台行为。
- 可以尽快把旧脚本中“真正有用但不危险”的部分迁进扩展。

主要涉及：
- DOM 适配层输出
- 站点上下文
- 少量扩展 UI

明确不做：
- fetch 拦截
- 保存链路
- 自动提交流程

首轮建议切口：
- `task-detail` 标注项只读识别与文本源/目标节点定位

原因：
- 它直接承接旧脚本中的 `getTargetTextarea()` 与 `getSourceText()` 这组低副作用能力。
- 它是后续快速填入、只读校验和轻量文本处理共同依赖的最小业务读入口。
- 它仍可保持完全只读，不会提前滑向自动化层。

当前状态：
- 首轮已完成。
- `annotation-item-collector.js` 已能在 `task-detail` 页面返回 item 列表、`sourceText`、`hasTargetTextarea`、`targetText` 与 `selectedValidity`。
- item 快照仍独立于 `SitePageState`，没有把 item 级字段塞回页面状态对象。
- 只读校验、只读反馈、单条 textarea 最小写回、单条 validity 最小切换、单条快速填入、单条受控操作编排和单条业务策略层都已完成。

下一步建议切口：
- 更接近完整快速填入的中风险业务迁移评估

原因：
- 现有 item 快照、quickfill、validity writer、validator、feedback 和 apply runner 已经足够支撑单条本地动作链。
- 下一步更适合继续评估哪些更接近完整快速填入的中风险能力值得迁入，而不必提前进入保存链。

### 阶段 5：中风险业务能力迁移

目标：
- 迁移依赖页面上下文、校验逻辑和文本处理，但仍不直接碰保存链路的能力

为什么排在这里：
- 这类能力开始接近旧脚本核心价值，但仍可先保持“本地只读计算或待执行建议”。

主要涉及：
- DOM 适配层
- 站点上下文
- 文本处理模块

明确不做：
- 自动保存替代
- 自动提交 / 自动抢单
- 服务端写入

### 阶段 6：高风险链路最后迁移

目标：
- 最后再单独处理 fetch 拦截、保存链路、自动提交、批处理、服务端对接等高风险能力

为什么排在最后：
- 它们最容易影响平台原始行为。
- 这些能力必须建立在前面所有只读输入和模块边界稳定之后。

主要涉及：
- 请求链路
- 保存协议
- 后端参考逻辑
- 高副作用操作流

明确不做：
- 与前置只读层混做

## 当前最适合的下一步

下一轮最适合执行的任务是：
- 联调缺陷记录与修复支撑（首版）

原因：
- 当前单脚本扩展版主线功能、手动控制面板和总验收支撑层已经具备，真正缺的是“如何在真实 Edge 页面持续记录和解释联调问题”。
- 页面规则、状态对象、DOM 适配结果、control panel、action history、flow report 和 runtime debug 已经足够支撑真实页面联调收尾。
- 在继续扩任何业务前，先把缺陷记录模板和只读调试快照导出补齐，更能避免把 `clicked / applied / completed` 误读成“服务端最终成功”。

当前这一步明确做什么：
- 新增联调缺陷记录文档
- 新增只读调试快照导出模块
- 固定 control panel、history、flow report 与 debug snapshot 的联调用法

当前这一步明确不做什么：
- 不新增 writer / runner / policy
- 不增强保存链或提交链
- 不做 fetch 拦截、自动重试、自动化动作、批处理或复杂 UI 重构
