# Agent 协作流程

## 目标

适配“规划 Agent -> 执行 Agent”的两段式工作流，在当前独立项目根目录下持续推进扩展迁移，不依赖旧聊天上下文。

## 当前项目边界

执行和规划都必须以当前项目根目录为唯一基准。

当前边界：
- `docs/` 是项目文档区。
- `extension/` 是正式扩展代码区。
- `legacy-reference/` 是旧代码参考区。

默认规则：
- 可以读取 `legacy-reference/` 做对照。
- 不应把 `legacy-reference/` 的旧代码整段复制进 `extension/`。
- 不删除、移动、重命名当前项目已有文件，除非用户明确授权。

## 规划 Agent 每轮应输出什么

每轮规划至少输出：
- 本轮真实查看过的文件、目录和关键事实
- 当前范围内的结构化认知摘要
- 正式代码与参考代码的边界判断
- 本轮不应做的事情
- 下一轮最适合执行的单一任务
- 对执行边界、写入范围、验证方式的限制
- 一段可直接交给执行 Agent 的 prompt

如果范围较大，规划 Agent 优先输出：
- 结构识别
- 模块盘点
- 耦合点
- 风险点
- 分层拆分建议

## 执行 Agent 每轮必须遵守什么

执行 Agent 必须遵守：
- 只在本轮目标所需的最小范围内改动文件。
- 每轮先更新相关 `docs/`，再做代码改动。
- 不一次性迁移高风险保存链路。
- 不一次性迁移自动提交、自动抢单、批处理。
- 不整段复制 `legacy-reference/` 旧代码。
- 不为了“顺手完整”而扩大任务范围。

## 建议的单轮粒度

推荐单轮只做：
- 1 个明确目标
- 1 个明确写入范围
- 1 个明确验证方式

示例：
- 只做页面识别与路由观察
- 只做只读状态采集骨架
- 只做 content script 装配层拆分
- 只做共享存储层固化

## 如何避免一次性大改

- 先复制结构边界，不复制全部能力。
- 先让扩展稳定观察页面，再让它修改页面。
- 先建立模块边界，再迁移旧逻辑细节。
- 先落低副作用能力，再碰高风险动作。
- 先把文档写清楚，再让执行 Agent 落代码。

## 建议工作顺序

1. 规划 Agent 扫描当前项目结构并确认边界。
2. 规划 Agent 修正文档中的路径、目录关系和迁移规则。
3. 执行 Agent 在 `extension/` 内实现本轮最小变更。
4. 执行 Agent 只验证本轮目标，不顺手扩大战线。
5. 规划 Agent 再评估剩余边界和下一轮拆分点。

## 推荐的执行 Prompt 模板

```text
你现在是执行 Agent。

工作根目录：
- 当前项目根目录就是唯一基准

允许读取：
- docs/
- extension/
- legacy-reference/

允许写入：
- 本轮目标相关的 docs 文件
- extension/ 下与本轮目标直接相关的最小范围文件

禁止事项：
- 不删除、移动、重命名当前项目已有文件
- 不整段复制 legacy-reference/ 中的旧代码
- 不迁移自动保存拦截、手动保存重放、自动提交、自动抢单
- 不顺手做与本轮目标无关的结构重组

本轮目标：
- <填写单一明确目标>

执行要求：
- 先检查并更新相关 docs，再改代码
- 只做最小必要改动
- 所有判断都基于当前项目实际结构
- 如需参考旧逻辑，只把 legacy-reference/ 当作参考来源

交付要求：
- 说明实际修改了哪些文件
- 说明如何验证
- 说明剩余风险
- 说明下一步最自然的衔接点
```
