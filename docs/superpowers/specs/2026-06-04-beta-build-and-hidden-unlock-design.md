# Beta 构建与隐藏解锁设计

## 目标

本设计用于为当前 `v0.4.0` 周期补齐一套正式包 / beta 包并行的最小方案。

本轮固定目标：

- 正式包与 beta 包共用同一套主代码，不长期分叉维护。
- `脚本下载中心` 只展示正式包，不展示 beta 包。
- beta 包只通过“查看外部目录”获取，且默认只保留一个当前 beta 包。
- beta 包默认界面与正式包一致，不新增独立 beta 首页或 beta 路由。
- 只有 beta 包在触发隐藏交互并输入正确口令后，才在当前页面中增量显示 beta 平台、beta 脚本和 beta 服务器选项。
- 解锁状态允许保存在本地缓存中，便于持续调试。
- 正式包即使有人手动篡改本地缓存，也不能显示 beta 内容。
- beta 平台被禁用后，popup / 右上角命中提示也不得继续暴露该平台。

## 当前现状

当前仓库已经具备如下基础，但尚未形成正式 / beta 双通道方案：

1. `extension/options/` 已有公开功能面板、脚本详情、系统管理和下载中心等结构化入口。
2. 下载中心当前通过后端聚合正式发布产物，可公开展示最新版本和历史版本。
3. popup / 页面命中检测当前会根据页面 URL 与脚本启用状态展示当前平台和脚本入口。
4. 系统管理当前已经有“正式服务器 / 本机服务器”一类后端地址配置保存流程。

当前缺口在于：

- 平台和脚本缺少“公开 / beta”可见级别。
- 扩展缺少统一的发行通道概念，无法区分 `public` 与 `beta` 包。
- popup 的命中识别更偏向“URL 是否匹配”，还没有与“当前是否允许对用户可见”强绑定。
- 下载中心和外部目录缺少“正式可见、beta 不可见”的产物边界。

## 设计原则

- 不新增长期分叉分支；正式功能与 beta 功能继续在同一主线演进。
- 不新增独立 beta 页面；beta 功能只在现有页面中增量显示。
- 正式包对 beta 能力实行硬隔离，不能依赖“用户不懂得改缓存”这种弱前提。
- “平台可见”“平台可启用”“平台被命中后是否显示”三者必须使用同一套判断。
- 先控制平台和脚本是否存在，再决定是否渲染 UI；避免出现“页面里看不见，但 popup 仍泄露”的裂缝。
- 继续复用当前 options 的保存模式，不额外引入复杂权限系统或远程授权系统。

## 方案对比

### 方案一：同一正式包内隐藏 beta 能力

做法：

- 所有人安装同一份 CRX。
- 通过隐藏交互 + 口令显示 beta 功能。

缺点：

- beta 代码、beta 系统信息和 beta 服务器信息会跟随正式包分发。
- 即使 UI 默认不显示，也无法满足“正式包完全不带 beta 可见能力”的目标。

### 方案二：正式包 / beta 包双构建，beta 包内再做隐藏解锁

做法：

- 保持同一代码仓。
- 打包时区分 `public` 与 `beta` 两种发行通道。
- 正式包不显示也不激活任何 beta 能力。
- beta 包支持隐藏交互 + 口令解锁，并在当前页面中增量显示 beta 内容。

优点：

- 正式用户拿不到 beta 下载入口，也看不到 beta 功能。
- beta 包默认外观与正式版一致，只有知晓入口的人才能解锁。
- 后续新增 beta 平台时只需标记可见级别，不需要再造第二套 UI。

缺点：

- 需要为构建脚本、可见性过滤和 popup 命中判断补一层统一状态。

### 方案三：长期 beta 分支单独维护

做法：

- 在 `beta` 分支上单独维护内测平台和内测服务器。
- 需要内测时从该分支手工打包。

缺点：

- 长期维护成本高。
- `v0.4.0` 收尾期间容易产生回合并冲突。
- 同一能力一旦要转正，需要频繁做分支搬运。

### 结论

采用方案二：

- 双构建产物负责发行边界。
- beta 包内的隐藏解锁负责调试边界。

## 发行通道设计

扩展新增统一的发行通道概念：

- `public`
- `beta`

### public 包

- 参与正式下载中心展示。
- 可保留当前最新版和历史版本列表。
- 不允许显示 beta 平台、beta 脚本、beta 服务器。
- 不允许任何本地 `betaUnlocked` 缓存生效。

### beta 包

- 不进入 `脚本下载中心`。
- 只通过“查看外部目录”获取。
- 默认只保留一个当前 beta 包，不做历史版本列表。
- 安装后默认界面与正式包一致。
- 仅在本地解锁成功后增量显示 beta 能力。

## 解锁流程设计

beta 包采用“两段式解锁”：

1. 隐藏交互
2. 口令校验

### 隐藏交互

- 推荐继续复用 options 左上角 Logo、品牌位或版本号区域。
- 例如连续点击 `7` 次触发解锁弹层。
- 正式包中不挂载这段隐藏交互逻辑。

### 口令输入

- 隐藏交互成功后，再显示口令输入层。
- 只有口令正确，才允许写入解锁缓存。
- 不要求每次打开扩展都重新输入；本地缓存可持续生效。

### 本地缓存字段

建议统一保存到公共元信息中，例如：

- `settings.meta.releaseChannel`
- `settings.meta.betaUnlocked`
- `settings.meta.betaUnlockedAt`

约束：

- `releaseChannel` 由构建产物确定，不能由用户在前端改出 `beta`。
- `betaUnlocked` 只有在 `releaseChannel === "beta"` 时才允许生效。
- 正式包即使读到 `betaUnlocked=true`，也必须按未解锁处理。

### 退出解锁

beta 包解锁后，建议在现有页面中提供一个显式“退出内测模式”操作：

- 仅在已解锁态显示。
- 点击后清空 `betaUnlocked` 相关缓存。
- 立即恢复为与正式包一致的默认显示。

## 平台与脚本可见性设计

平台和脚本新增统一可见级别：

- `public`
- `beta`

### 公开平台 / 脚本

- 在正式包和 beta 包中都可见。
- 是否展示仍由脚本启停状态、当前路由和现有业务规则决定。

### beta 平台 / 脚本

- 正式包中永远不可见。
- beta 包未解锁时不可见。
- beta 包已解锁时才可见。

### Lightwheel 作为 beta 平台的预期

- 正式包：功能面板不显示 `Lightwheel`，详情入口不可达。
- beta 包未解锁：同样不显示 `Lightwheel`。
- beta 包已解锁：`Lightwheel` 出现在当前功能面板的平台列表中，脚本与详情页行为和普通平台一致。

## 页面增量显示设计

本轮不新增 beta 专属页面，不新增 beta 专属路由。

beta 包解锁后只在当前页面中追加内容，例如：

- 功能面板平台列表中追加 beta 平台
- 平台摘要中追加 beta 平台的默认启用脚本
- 脚本详情列表中允许进入 beta 脚本配置
- 系统管理的服务器选项中追加 beta 服务器

这意味着用户的感知应为：

- 默认界面像正式版
- 解锁后，现有页面里多出一批新平台、新脚本和新服务器选项

## 命中检测与 popup 可见性设计

命中检测不能只依据 URL 是否匹配，必须同时满足“当前用户真正可使用该平台”。

统一有效可见状态建议为：

- 当前发行通道允许该平台存在
- 当前解锁态允许该平台存在
- 当前平台或脚本在本地配置中处于启用状态

只有三者都满足时，popup / 右上角卡片才允许显示：

- `当前页面属于 <平台>`
- 平台名称 pill
- 进入脚本设置按钮
- 相关命中说明文案

### Lightwheel 的命中示例

- 正式包：即使打开 `label-cloud.lightwheel.net`，也不得显示 `Lightwheel` 命中信息。
- beta 包未解锁：同样不得显示 `Lightwheel` 命中信息。
- beta 包已解锁但 `Lightwheel` 被禁用：同样不得显示 `Lightwheel` 命中信息。
- beta 包已解锁且 `Lightwheel` 启用：才允许显示 `Lightwheel` 命中信息。

这套规则必须同时覆盖：

- popup 平台命中摘要
- popup 中的脚本入口按钮
- “打开当前脚本设置”按钮
- 页面右上角浮层的命中文案
- 与平台命中直接相关的辅助说明

## 服务器选项设计

系统管理现有后端地址配置继续保留在原页面中，不拆新页面。

默认服务器候选项：

- `正式服务器`
- `本机服务器`

仅当 `beta` 包已解锁时，再追加：

- `Beta 服务器`

### 显示规则

- 正式包：只显示 `正式服务器 / 本机服务器`
- beta 包未解锁：只显示 `正式服务器 / 本机服务器`
- beta 包已解锁：显示 `正式服务器 / 本机服务器 / Beta 服务器`

### 保存规则

- 继续沿用当前“选择后点击保存按钮才落本地缓存”的模式。
- 不额外新增后台认证流程。
- `Beta 服务器` 的实际地址建议与现有正式 / 本机服务器配置一样，由统一设置入口维护。

## 下载中心与外部目录设计

### 脚本下载中心

- 只展示 `public` 产物。
- 继续保留最新版突出展示与历史版本下拉选择。
- 不展示 beta 包，不给 beta 包占位，不给 beta 包留“实验版”提示。

### 查看外部目录

- 继续保留作为兜底入口。
- beta 包只通过该入口对应的外部目录提供下载。
- 外部目录中 beta 包默认只保留一个当前文件，不做历史版本切换。

## 模块边界建议

本轮实现建议至少收口以下模块边界：

### 共享发布态

职责：

- 提供当前构建的 `releaseChannel`
- 读取并校验 `betaUnlocked`
- 输出统一的“beta 能否生效”结论

### 可见性过滤

职责：

- 根据 `releaseChannel + betaUnlocked + enabled state` 过滤平台与脚本
- 向 options、popup 和命中检测提供统一结果

### 构建产物配置

职责：

- 决定当前打包是 `public` 还是 `beta`
- 决定下载中心是否纳入该产物
- 决定是否写入 beta 相关元信息

### 系统管理服务器候选项

职责：

- 基于统一可见性状态决定是否显示 `Beta 服务器`
- 继续复用现有保存流程

## 非目标

本轮不做：

- 不新增远程鉴权或在线授权系统
- 不新增独立 beta 首页或 beta 路由
- 不要求正式包和 beta 包在同一浏览器内同时安装
- 不改现有平台业务逻辑本身，只补平台可见性和入口控制
- 不把 beta 包加入公开下载中心

## 最低验收标准

- 正式包中看不到 `Lightwheel` 平台、脚本入口和命中提示。
- beta 包未解锁时，也看不到 `Lightwheel` 平台、脚本入口和命中提示。
- beta 包解锁后，功能面板中出现 `Lightwheel` 平台与脚本入口。
- beta 包解锁后，系统管理里出现 `Beta 服务器` 选项。
- beta 包解锁后，如果用户禁用 `Lightwheel`，popup / 右上角命中提示不得继续显示 `Lightwheel`。
- 清除解锁状态后，beta 增量内容再次消失。
- 正式包即使人为写入 `betaUnlocked=true` 缓存，也不得显示任何 beta 内容。

## 实施顺序建议

建议按下面顺序推进：

1. 补发行通道与打包产物区分
2. 补共享发布态与 `betaUnlocked` 校验
3. 给平台 / 脚本补 `public | beta` 可见级别
4. 收口 options 功能面板、详情页与系统管理的可见性过滤
5. 收口 popup / 命中检测对统一可见状态的复用
6. 最后处理下载中心与外部目录的产物边界

原因：

- 先把“能不能存在”定清，再去改各个页面显示，否则很容易漏 popup 和命中卡。

## 测试策略

### 前端

- 正式包加载后，确认 beta 平台在功能面板、详情入口和 popup 中都不可见。
- beta 包未解锁时，确认显示与正式包一致。
- beta 包解锁后，确认 beta 平台与 beta 服务器选项按预期出现。
- beta 平台禁用后，确认命中提示立即消失。
- 清除解锁态后，确认 beta 内容重新隐藏。

### 构建与发布

- `public` 构建应继续进入下载中心。
- `beta` 构建不进入下载中心。
- 外部目录中的 beta 包仅保留一个当前文件。

### 回归

- 公开平台与公开脚本的既有启停、详情入口、服务器切换能力不应受影响。
- popup 现有已启用平台的命中识别逻辑不应因为 beta 过滤而回退。

3. options 可见性过滤
4. popup / 命中检测过滤
5. 下载中心与外部目录边界
